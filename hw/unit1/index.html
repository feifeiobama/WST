Unit 1: The Overview and fundamentals

Homework 1

For the three categories of web software technology we've just defined, you are asked to find out which category so-called "Chrome plug-in" technology belongs to. Please describe the basic idea of the technology, and give your reasoning on its position in the technology categories.

I think "Chrome plug-in" technology belongs to Distributed Object Technology.

"Chrome plug-in" is a kind of plugin that runs within a web page and adds functionality to it by invoking PPAPI (or NPAPI on older Chrome versions). For example, Adobe Flash Player in Chrome is a PPAPI plugin that helps view multimedia contents and execute Internet applications.

"Chrome plug-in" follows the principle of "distributed object". It is distributed on browser side and gives users remote access to objects or functions on a different server. Further more, PPAPI allows it to "plug and play" anywhere. These makes it interoperable regardless of time, space, platform and awareness, and that's why it belongs to Distributed Object Technology.

Homework 2

In a normal intercommunication between Web browser and Web Server, Web Server closes the network communication right after it sends back the document requested by the browser. Think about this statusless character of World Wide Web. Give some good points and bad points of it. Can a web server know exactly how many browsers are reading its document?

Good points of the statusless character of World Wide Web:

  • Without having to wait ages for client to respond, web server can be more efficient, which means it can response quicker and handle more requests.
  • Statusless makes logic simpler for both server end and client end, and therefore reduces cpu and memory consumption.
  • This also relieves pressure on the network line of both sides.

Bad points:

  • Statusless character is not conductive to complex communication between server and client.
  • The server doesn't know if the client have received the whole document, and the repeated transmission of a full document can be a great waste.
  • The cost of reconnecting is relatively high when a client requests many documents from the same website.

A web server CANNOT know exactly how many browsers are reading its document in a normal intercommunication case, because the server closes communication right after it sends out the document, and browsers won't inform the server when closing the document.

Homework 3 (optional)

Open Question: of those client side technologies, why Java Applet is getting dead and Javascript success? Try to explain from the "REST" perspective.

To run Java applet you'll need to install JRE, which is inconvenient. Portability is also limited since some mobile platforms don't support applet runs. JavaScript, on the contrary, is web native and plays nice with HTML and CSS. It is widely supported by most modern explorers.

Java applet runs within JVM in a process separate from the web browser itself, so its capability to interact with browser is not as good as JavaScript. Since it is self-contained, it doesn't have access to client-side resources like files and operating system.

There are many other reasons. For example, Java applet is much unsafer, since malicious developers may implant backdoors to track users or perform harmful actions. Besides, applets must be compiled into machine code before testing or deploying, while you can simply run JavaScript codes. There is also saying that JavaScript is easier to learn.

In a word, Java applet violets some constraints of the REST architecture style but JavaScript doesn't. That is why Java Applet is getting dead and Javascript success.
